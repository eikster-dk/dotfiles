Omarchy Auto Nightlight

This page explains the script at ~/.local/bin/omarchy-auto-nightlight, the user systemd timer that runs it, and the Hypridle hooks that reapply it after sleep and DPMS resume.

Data Source

wttr.in provides today’s sunrise and sunset as JSON.

Nightlight Engine

hyprsunset applies the screen temperature through Hyprland.

Manual Override

If you change nightlight manually, automation pauses for the rest of the day.

Runtime Flow

What Happens When The Script Runs

1. Load configRead temperatures and optional coordinates.
2. Check overrideExit if manual override is active today.
3. Get sun modeUse cached or fresh wttr.in sunrise/sunset.
4. Start engineEnsure hyprsunset is running.
5. Apply tempSet night or day temperature and save state.
Configuration

Defaults And User Config

The script starts by defining paths and defaults. It uses XDG locations when present, otherwise it falls back to ~/.config and ~/.cache.

CONFIG_FILE="$CONFIG_HOME/omarchy/auto-nightlight.conf"
CACHE_DIR="$CACHE_HOME/omarchy/auto-nightlight"
SUN_CACHE="$CACHE_DIR/sun.env"
STATE_FILE="$CACHE_DIR/state.env"

NIGHT_TEMP=4000
DAY_TEMP=6000
SUN_OFFSET_MINUTES=0

Then it sources ~/.config/omarchy/auto-nightlight.conf if the file exists. That lets you override temperatures, the sunrise/sunset offset, or set exact coordinates.

If LATITUDE and LONGITUDE are configured, the script asks wttr.in for that exact location. If not, wttr.in infers your approximate location from your public IP.

Failure Handling

Notifications When Work Fails

The fail path calls notify-send with critical urgency, then prints the same message to stderr for systemd logs.

notify-send --urgency=critical "Nightlight failed" "$message"
printf 'omarchy-auto-nightlight: %s\n' "$message" >&2

This means failures should be visible both as desktop notifications and in journalctl --user -u omarchy-auto-nightlight.service.

wttr.in

Fetching Sunrise And Sunset

The script builds one of two URLs:

  • https://wttr.in/?format=j1 when coordinates are not configured.
  • https://wttr.in/<latitude>,<longitude>?format=j1 when coordinates are configured.

It reads these fields from the JSON response with jq:

.weather[0].date
.weather[0].astronomy[0].sunrise
.weather[0].astronomy[0].sunset
.request[0].query
.nearest_area[0].areaName[0].value

The date and sunrise/sunset strings are combined and parsed with date -d, then converted into epoch seconds. That makes later comparisons simple numeric checks.

Caching

Daily Sun Cache

The script avoids calling wttr.in every five minutes. It writes a daily cache to ~/.cache/omarchy/auto-nightlight/sun.env.

SUN_CACHE_DATE=2026-05-10
SUN_CACHE_WTTR_LOCATION=55.783,12.367
SUN_CACHE_WTTR_QUERY='Lat 55.78 and Lon 12.38'
SUN_CACHE_WTTR_AREA='Søndergården'
SUNRISE_EPOCH=1778382540
SUNSET_EPOCH=1778440020

The cache is considered fresh only when it matches today’s date and the configured wttr.in location. If either changes, the script fetches new data.

Day Or Night

Choosing The Target Temperature

current_sun_mode compares the current epoch time against the cached sunrise and sunset epochs.

if ((now < SUNRISE_EPOCH || now >= SUNSET_EPOCH)); then
  printf 'night\n'
else
  printf 'day\n'
fi

Night mode maps to NIGHT_TEMP. Day mode maps to DAY_TEMP. By default those are 4000 and 6000.

Manual Override

Respecting Your Manual Toggle

The script stores what it last applied in ~/.cache/omarchy/auto-nightlight/state.env.

LAST_APPLIED_DATE=2026-05-10
LAST_APPLIED_TEMP=4000

On the next run, it asks hyprsunset for the current temperature. If the current value differs from LAST_APPLIED_TEMP, the script assumes you manually toggled or overrode nightlight.

MANUAL_OVERRIDE_DATE=2026-05-10

Once that date is recorded, every later timer run exits without changing anything until the next local day.

This is why manually running omarchy-toggle-nightlight should not be undone by the timer five minutes later.

hyprsunset

Applying The Temperature

Before setting the temperature, the script ensures hyprsunset is running. It starts it through uwsm-app when available, matching Omarchy’s own command style.

setsid uwsm-app -- hyprsunset &

Then it applies the target temperature directly:

hyprctl hyprsunset temperature "$target_temp"

This is intentionally different from omarchy-toggle-nightlight. A timer needs idempotent behavior: at night it should set night temperature, not toggle back and forth.

systemd

How The Timer Runs It

The user timer starts omarchy-auto-nightlight.service. The service runs the script once and exits.

[Timer]
OnBootSec=30s
OnUnitActiveSec=5min
Persistent=true
  • OnBootSec=30s runs it shortly after your user session starts.
  • OnUnitActiveSec=5min runs it every five minutes after the last run.
  • Persistent=true catches up after missed runs, such as boot or resume.
Hypridle

Sleep And DPMS Hooks

hypridle.conf calls omarchy-auto-nightlight apply after sleep and when DPMS resumes.

The script does not run when DPMS turns off. There is no benefit to changing screen temperature while the display is powered down, and avoiding display-affecting work at that moment reduces the chance of waking the display back up.

Operations

Useful Commands

systemctl --user status omarchy-auto-nightlight.timer
systemctl --user list-timers omarchy-auto-nightlight.timer
systemctl --user start omarchy-auto-nightlight.service
journalctl --user -u omarchy-auto-nightlight.service -n 50

To clear cached sun data or manual override state, remove files from ~/.cache/omarchy/auto-nightlight/.
